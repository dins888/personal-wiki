## 定义

**OpenSpec** 是由 Fission AI 主导开发的一款轻量级、开源的**规格驱动开发**（Spec-Driven Development, SDD）框架。它的核心理念是"规格先行"——在 AI 编程助手编写任何代码之前，先让人类与 AI 就需求和规格达成共识，从而锁定开发意图，避免 AI 在编码过程中偏离需求。

OpenSpec 无需 API 密钥或 MCP（Model Context Protocol）集成即可运行，通过在代码仓库中组织规格文件来实现工作流管理。它还提供了一个可选的 MCP Server，用于与更多 AI 助手集成。

OpenSpec 的最大特色是**棕地优先**（Brownfield-first）策略——专为已有代码库设计，允许开发者在不破坏现有工作流的前提下引入规格驱动开发，这与许多从零开始（绿地，Greenfield）的工具形成鲜明对比。


## 第一性

> 用结构化模板把模糊需求转化为可验证规格的方法论。

## 比方

> 像建筑师在施工前画出的精确蓝图——业主说「我想要一栋温馨的房子」，蓝图把它翻译成尺寸、材料、承重数据，工人照图施工，验收有据可查。

## 核心功能与特性

### 1. 规格先行工作流（Spec-First Workflow）

OpenSpec 将开发过程组织为四个核心阶段：

| 阶段 | 命令 | 核心动作 | 输出产物 |
|------|------|---------|---------|
| 提案（Proposal） | `/opsx:new` | 捕获变更意图，明确"做什么、为什么" | `proposal.md` |
| 快进生成（Fast-Forward） | `/opsx:ff` | 自动生成所有规划文档 | `specs/`、`design.md`、`tasks.md` |
| 实施（Apply） | `/opsx:apply` | AI 依据规格和任务列表编写代码 | 源代码变更 |
| 归档（Archive） | `/opsx:archive` | 归档已完成的变更，合并规格更新 | 更新后的 `specs/` |

### 2. 棕地优先设计（Brownfield-First）

OpenSpec 是专门为**已有代码库**设计的规格驱动工具。它将"真实来源"（`openspec/specs/`，当前系统行为）与"变更提案"（`openspec/changes/`，提议的更新）在文件系统上物理分离，确保变更的原子性和可审计性。

### 3. 广泛的 AI 工具兼容性

OpenSpec 原生支持 20+ 种 AI 编程助手，包括：

| 工具 | 集成方式 |
|------|---------|
| Claude Code | 原生斜杠命令 |
| Cursor | 原生斜杠命令 |
| GitHub Copilot | `.github/prompts/` 目录 |
| Amazon Q Developer | `.amazonq/prompts/` 目录 |
| Windsurf | `.windsurf/workflows/` 目录 |
| 其他（RooCode、Cline 等） | 通过 `AGENTS.md` 文件 |

### 4. 零配置启动

- 无需 API 密钥
- 无需 MCP 集成（可选）
- 通过 npm 全局安装，5 分钟内完成初始化
- 无需 Python 环境

### 5. Markdown 驱动的存储

OpenSpec 拒绝复杂的数据库依赖，采用 Markdown 文件作为存储方案。规格文件与源代码一同存放在 Git 仓库中，成为"活文档"（Living Documentation），可随代码一起版本控制和审查。

## 常见用法示例

### 一、安装与初始化

```bash
# 全局安装 OpenSpec CLI
npm install -g @fission-ai/openspec@latest

# 在项目目录中初始化
cd my-project
openspec init
```

### 二、典型开发工作流

```
开发者：/opsx:new add-dark-mode
AI：    Created openspec/changes/add-dark-mode/
        Ready to create: proposal

开发者：/opsx:ff
AI：    ✓ proposal.md — why we're doing this, what's changing
        ✓ specs/       — requirements and scenarios
        ✓ design.md    — technical approach
        ✓ tasks.md     — implementation checklist
        Ready for implementation!

开发者：/opsx:apply
AI：    Implementing tasks...
        ✓ 1.1 Add theme context provider
        ✓ 1.2 Create toggle component
        ✓ 2.1 Add CSS variables
        ✓ 2.2 Wire up localStorage
        All tasks complete!

开发者：/opsx:archive
AI：    Archived to openspec/changes/archive/2026-07-09-add-dark-mode/
        Specs updated. Ready for the next feature.
```

### 三、规格文件结构

```
openspec/
├── specs/                    # 真实来源（当前系统行为）
│   ├── auth-login/
│   │   └── spec.md
│   ├── auth-session/
│   │   └── spec.md
│   ├── checkout-cart/
│   │   └── spec.md
│   └── checkout-payment/
│       └── spec.md
├── changes/                  # 活跃的变更提案
│   └── add-dark-mode/
│       ├── proposal.md
│       ├── specs/
│       │   └── auth-session/
│       │       └── spec.md   # 规格增量
│       ├── design.md
│       └── tasks.md
└── archive/                  # 已完成的变更
```

## 相关概念

- **规格驱动开发（Spec-Driven Development, SDD）**：OpenSpec 所归属的开发方法论，强调在编码前先完成并确认规格文档，以规格作为开发和验证的唯一标准。
- **Spec Kit**：GitHub 出品的类似工具包，与 OpenSpec 同属 SDD 工具，但更重量级、采用严格的阶段门控、适合绿地项目。OpenSpec 官方将其定位为更轻量的替代品。
- **Kiro**：AWS 推出的规格驱动 IDE，功能强大但锁定特定 IDE 和 Claude 模型。OpenSpec 强调无厂商锁定。
- **BMAD Method**：多代理系统框架，适用于企业级多仓库项目，设置复杂度高。OpenSpec 专注于变更管理，设置仅需 5 分钟。
- **AGENTS.md**：OpenSpec 在项目中生成的指令文件，让任何能读取 Markdown 的 AI 工具都能使用 OpenSpec 工作流。
- **MCP（Model Context Protocol）**：OpenSpec 提供可选的 MCP Server 实现，使更多 MCP 兼容的 AI 助手可以直接与 OpenSpec 交互。
- **棕地开发（Brownfield）与绿地开发（Greenfield）**：棕地指在已有代码库上开发，绿地指从零开始构建。OpenSpec 专为棕地设计，Spec Kit 更适合绿地。
