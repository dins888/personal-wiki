## 定义

**Honcho** 是一个 AI 原生的跨会话记忆后端（Memory Provider），为 [Hermes Agent](/entries/tech/hermes/) 提供持久化用户建模和深度记忆能力。它替代了 Hermes 内置的简单文件存储（MEMORY.md / USER.md），在服务端维护一个关于用户"是谁"的动态模型——偏好、沟通风格、目标、行为模式——并在对话结束后通过 **辩证推理（Dialectic Reasoning）** 持续更新这个模型。

Honcho 由 [Nous Research](https://nousresearch.com/) 开发，支持云端托管（app.honcho.dev）或本地自托管。官网：[honcho.dev](https://honcho.dev)。


## 第一性

> 不是记住用户说了什么，而是推理出用户是什么样的人——从对话中提炼，跨会话累积。

## 比方

> 一个极其细心的私人助理——每次对话结束后都在小本子上记录"老板喜欢什么、讨厌什么、工作习惯如何"，下次见面时已经准备好了你需要的背景信息，而不是每次都从零开始了解你。

**命名说明：**

"Honcho" 在英语俚语中意为"头儿、负责人"。作为项目名，它暗示了这个系统对用户记忆拥有"管理权"——它负责记住你是谁、你需要什么。

## 核心概念

### 记忆提供者插件（Memory Provider Plugin）
Honcho 以插件形式集成到 Hermes 的 Memory Provider 系统中，替代内置的文件存储。通过 `hermes memory setup` 选择 "honcho" 即可启用。

### 双对等体架构（Dual-Peer Architecture）
Honcho 将对话建模为两个"对等体"之间的交互：
- **用户对等体（User Peer）**：从用户消息中学习，构建用户表征（偏好、习惯、目标）
- **AI 对等体（AI Peer）**：从 AI 消息中学习，构建 AI 自身的知识表征

每个 Hermes 配置文件拥有独立的 AI Peer，多个 Agent 同时服务同一用户时，各自的观察和结论互不干扰。

### 辩证推理（Dialectic Reasoning）
Honcho 的核心差异化能力。对话发生后，Honcho 用 LLM 对用户交互进行分析，提炼出关于用户的洞察（偏好、习惯、行为模式）。这些洞察随时间累积，形成对用户日益加深的理解。

支持多轮辩证深度（1-3 轮）：
- **第 0 轮**：冷启动（通用查询）或热启动（会话范围查询）
- **第 1 轮**：自我审计——识别初始评估中的漏洞
- **第 2 轮**：调和——检查矛盾，生成最终综合

### 两层上下文注入
每轮对话中，Honcho 向系统提示注入两层上下文：
1. **基础上下文**：会话摘要 + 用户表征 + 用户卡片 + AI 表征 + AI 身份卡片 → "用户是谁"
2. **辩证补充**：LLM 合成的关于用户当前状态和需求的推理 → "当前什么重要"

### 会话策略（Session Strategy）
控制消息和观察记录的作用域范围：

| 策略 | 行为 |
|------|------|
| per-directory（默认） | 每个工作目录一个会话 |
| per-repo | 每个 Git 仓库根目录一个会话 |
| per-session | 每次 Hermes 运行一个新会话 |
| global | 所有目录共享一个会话 |

## Honcho 提供的 5 个工具

Hermes 暴露给 Agent 的 Honcho 工具集：

| 工具 | 功能 |
|------|------|
| `honcho_profile` | 快速检索对等体卡片（不调用 LLM），返回关于用户的关键事实 |
| `honcho_search` | 语义搜索记忆，返回按相关性排序的原始摘录 |
| `honcho_context` | 辩证 Q&A，由 Honcho 的 LLM 综合对话历史生成答案 |
| `honcho_conclude` | 当用户表达偏好、纠正或重要上下文时，将持久化事实写入 Honcho |
| `honcho_status` | 显示当前 Honcho 配置、连接测试和对等体信息 |

## Honcho vs 内置记忆

| 能力 | 内置记忆 | Honcho |
|------|---------|--------|
| 跨会话持久化 | ✔ 文件 MEMORY.md/USER.md | ✔ 服务端 API |
| 用户画像 | ✔ Agent 手动维护 | ✔ 自动辩证推理 |
| 会话摘要 | ✘ | ✔ 自动注入 |
| 多 Agent 隔离 | ✘ | ✔ 按 Peer 分离 |
| 观察模式 | ✘ | ✔ 统一/定向观察 |
| 衍生洞察 | ✘ | ✔ 服务端推理 |
| 跨历史搜索 | ✔ 全文检索 FTS5 | ✔ 语义搜索 |

## 召回模式（Recall Modes）

| 模式 | 自动注入上下文 | 工具可用 | 适用场景 |
|------|--------------|---------|---------|
| hybrid（默认） | ✔ | ✔ | Agent 自主决定何时用工具、何时依赖自动上下文 |
| context | ✔ | ✘ | 仅需自动注入，不暴露工具 |
| tools | ✘ | ✔ | 仅通过工具按需检索 |

## 快速配置

```bash
# 交互式设置向导
hermes memory setup    # 选择 "honcho"

# 云端托管
hermes honcho setup    # 选择 "cloud"，粘贴 API Key

# 本地自托管
hermes honcho setup    # 选择 "local"，输入 http://localhost:8000

# 验证
hermes honcho status   # 查看配置、连接状态、Peer 信息
```

## 关键配置旋钮

| 旋钮 | 控制项 | 默认值 |
|------|--------|--------|
| `contextCadence` | 基础上下文刷新间隔（轮数） | 1 |
| `dialecticCadence` | 辩证推理 LLM 调用间隔（轮数） | 2 |
| `dialecticDepth` | 每次辩证推理的轮数（1-3） | 1 |

三个旋钮相互正交——可以频繁刷新上下文但少用辩证，也可以低频深度多轮辩证。

## 相关概念

- **Hermes Agent**：Honcho 的宿主 Agent 框架，由 Nous Research 开发
- **MEMORY.md / USER.md**：Hermes 内置的简单文件记忆，Honcho 替代/增强之
- **Dialectic Reasoning（辩证推理）**：Honcho 的核心差异化能力
- **Peer（对等体）**：Honcho 中用户和 AI 各自的身份表征
- **Context Engine**：Hermes 的上下文引擎，Honcho 是其记忆提供者之一
