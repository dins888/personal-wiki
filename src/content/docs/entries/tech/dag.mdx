## 定义

**DAG**（Directed Acyclic Graph，有向无环图）是图论中的一种基本数据结构，由有限个**顶点**（Vertex）和**有向边**（Directed Edge）组成，且图中不存在任何**环**（Cycle）——即沿着有向边的方向遍历，无法回到起始顶点。

DAG 是介于一般**有向图**（允许环存在）和**树**（要求任意两节点间仅有唯一路径）之间的图结构。它既允许多个父节点（区别于树），又禁止循环依赖（区别于一般有向图），这使得 DAG 在表达"偏序关系"和"依赖关系"时具有不可替代的优势。

DAG 在计算机科学中有极其广泛的应用，包括任务调度、编译器优化、版本控制、区块链共识等领域。

## 核心功能与特性

DAG 具有以下关键特性，使其在各类工程场景中被广泛采用：

### 1. 拓扑排序（Topological Ordering）

DAG 中的顶点可以进行**拓扑排序**，即排列成一个线性序列，使得对于每条有向边 (u, v)，顶点 u 都排在 v 之前。这种排序天然表达了"依赖优先级"，是任务调度的理论基础。

```python
# 拓扑排序示例（Kahn 算法）
from collections import deque

def topological_sort(vertices, edges):
    in_degree = {v: 0 for v in vertices}
    for u, v in edges:
        in_degree[v] += 1
    queue = deque([v for v in vertices if in_degree[v] == 0])
    result = []
    while queue:
        node = queue.popleft()
        result.append(node)
        for neighbor in adj[node]:
            in_degree[neighbor] -= 1
            if in_degree[neighbor] == 0:
                queue.append(neighbor)
    return result
```

### 2. 依赖关系的自然表达

DAG 天然适合建模"先决条件"和"依赖关系"：

- **任务 A 依赖任务 B**：用一条从 B 指向 A 的有向边表示
- **不可循环**：若 A 依赖 B、B 依赖 C，则 C 不可能反向依赖 A（否则形成死锁）
- **多对多关系**：一个任务可以有多个前置依赖，也可以被多个后续任务依赖（区别于树结构）

### 3. 无环性与终止保证

DAG 的"无环"特性保证了：

- 任何沿边方向的遍历最终会终止（不会无限循环）
- 不存在死锁或循环依赖
- 可以通过简单的 DFS/BFS 进行**环检测**

### 4. 高效的路径分析

DAG 上可以高效完成最短路径、关键路径（Critical Path）等分析，在项目管理（如 PERT 图）和编译器优化中至关重要。

### 5. 记忆化与去重

在递归计算中，DAG 结构天然支持**记忆化**（Memoization）——共享子表达式，避免重复计算。这也是动态规划算法的本质图结构。

## 常见用法示例

### 一、区块链中的 DAG 共识

DAG 被用作替代传统区块链（线性链式结构）的分布式账本架构。与传统区块链不同，DAG 型区块链允许交易并发处理，每个新交易需要验证若干旧交易，形成网状拓扑而非线性链。

```
传统区块链（线性结构）：
  Block 1 → Block 2 → Block 3 → Block 4 → Block 5

DAG 结构（网状结构）：
    Tx1 ──→ Tx3 ──→ Tx5
    │        ↑       ↑
    └→ Tx2 ─┘   ─→ Tx4 ─┘
```

**代表项目：**

| 项目 | 特点 |
|------|------|
| IOTA (Tangle) | 最早的 DAG 区块链，每个交易需验证两个前驱交易 |
| Nano | 使用 Block Lattice 结构，每个账户有独立链 |
| Fantom | 采用 Lachesis 共识协议的 DAG 架构 |

**核心优势：**

- 并发处理交易，理论上 TPS（Transactions Per Second）可扩展性更高
- 无需矿工，交易参与者自身即为验证者
- 无区块确认等待时间，适合微支付和物联网场景

### 二、任务调度（Apache Airflow、CI/CD Pipeline）

DAG 是任务调度系统中最核心的概念。以 **Apache Airflow** 为例，一个工作流（Workflow）就是一个 DAG，其中的每个节点代表一个任务，有向边表示任务之间的依赖关系。

```python
# Apache Airflow DAG 定义示例
from airflow import DAG
from airflow.operators.python import PythonOperator
from datetime import datetime

def extract():
    print("Extracting data...")

def transform():
    print("Transforming data...")

def load():
    print("Loading data...")

with DAG(
    dag_id="etl_pipeline",
    start_date=datetime(2026, 1, 1),
    schedule_interval="@daily",
) as dag:

    extract_task = PythonOperator(task_id="extract", python_callable=extract)
    transform_task = PythonOperator(task_id="transform", python_callable=transform)
    load_task = PythonOperator(task_id="load", python_callable=load)

    # 定义依赖关系：extract → transform → load
    extract_task >> transform_task >> load_task
```

**CI/CD Pipeline 中的 DAG：**

```yaml
# GitHub Actions 示例：多个 Job 的 DAG 依赖
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Running linter..."

  test:
    needs: lint        # test 依赖 lint 完成
    runs-on: ubuntu-latest
    steps:
      - run: echo "Running tests..."

  build:
    needs: [lint, test]  # build 依赖 lint 和 test 都完成
    runs-on: ubuntu-latest
    steps:
      - run: echo "Building..."

  deploy:
    needs: build          # deploy 依赖 build 完成
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying..."
```

上述 CI/CD 流水线形成一个 DAG：

```
lint → test → build → deploy
  └──────────────↗
```

### 三、编译器中的依赖分析

在编译器后端优化中，DAG 用于表示**基本块**（Basic Block）内的指令依赖关系，称为 **DAG 表示**或**有向无环图表示**：

```
源代码：
  a = b + c
  d = b + c    // 与 a 的计算完全相同
  e = a + d

对应的 DAG：
       (+)
      /   \
    (b)   (c)
     │     │
     a     d       // a 和 d 共享同一子表达式节点
      \   /
       (+)
        │
       (e)
```

**关键用途：**

- **公共子表达式消除**（Common Subexpression Elimination）：识别并合并相同的子表达式节点
- **指令调度优化**：通过拓扑排序确定指令执行顺序
- **死代码消除**：不可达节点可安全删除

### 四、版本控制中的 Git

Git 的版本历史本质上是一个 DAG：

- 每个 **commit** 是 DAG 中的一个节点
- 每个 commit 指向其**父 commit**（有向边），形成从子到父的依赖关系
- **merge commit** 可以有多个父节点（多分支合并）
- 整个仓库的 commit 历史构成一个 DAG，而非线性链表

```
Git commit DAG 示意图：

  A → B → C → D (main)
           ↘
             E → F (feature)
                 ↘
                   G → H (release, merge of D and F)
```

- 拓扑排序在 Git 中的体现：`git log --topo-order`
- **环检测**保证了 Git 历史不可能出现循环引用

## 相关概念

- **拓扑排序（Topological Sort）**：将 DAG 的顶点排列成线性序列，使得每条边的起点都在终点之前。DAG 最核心的算法操作。
- **图论（Graph Theory）**：DAG 所属的数学分支，研究顶点与边构成的图结构的性质。
- **树（Tree）**：一种特殊的 DAG，要求每个节点最多只有一个父节点（根节点无父节点）。DAG 是树的推广，树是 DAG 的特例。
- **有向图（Directed Graph）**：边有方向的图，DAG 是有向图加上"无环"约束后的子集。
- **环检测（Cycle Detection）**：判断图中是否存在环的算法。DAG 的定义要求无环，因此环检测是验证一个有向图是否为 DAG 的关键操作。常用 DFS 着色法或 Kahn 算法。
- **任务调度（Task Scheduling）**：DAG 最经典的应用场景之一。操作系统进程调度、分布式计算任务编排、CI/CD 流水线等均基于 DAG 建模。
- **区块链（Blockchain）**：传统区块链是 DAG 的退化形式（每个节点只有一个子节点），DAG 型区块链是其推广。
- **Git**：分布式版本控制系统，其 commit 历史是一个典型的 DAG 结构。
