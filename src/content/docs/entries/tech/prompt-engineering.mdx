## 定义

提示工程（Prompt Engineering）是一门系统性地设计、优化和迭代提示词（Prompt）的方法论，目标是在不修改模型参数的前提下，通过精心构造输入指令来引导大语言模型（LLM）生成更准确、更符合预期的输出。通俗来说，如果你把 AI 比作一个能力极强但需要明确指示的助手，提示工程就是研究"怎么说，AI 才能听得懂、做得好"的学问。


## 第一性

> 通过精心设计自然语言指令来引导 AI 模型产出期望结果的技术。

## 比方

> 像给一位能力极强但不懂你上下文的实习生写任务说明书——写得越精准、示例越清晰，交付物就越接近你要的东西。

## 英文术语拆解

|--------|----------|------|----------|------------------|
| Prompt | /prɑːmpt/ | 名词 | 提示词、指令 | 指输入给模型的引导性文本 |
| Engineering | /ˌendʒɪˈnɪrɪŋ/ | 名词 | 工程、工程学 | 强调这是一种系统化的、可复现的工程实践，而非随意的"试错" |

### 组合理解

"Prompt"与"Engineering"的组合，强调提示词的设计不是随口一问，而是一套**可复现、可迭代、可评估**的工程方法。它将"跟 AI 说话"从一种经验性的艺术，转变为一门有方法论支撑的技术——就像软件工程将"写代码"从手艺变成了工程一样。

## 详细说明

### 核心原则

提示工程遵循五项核心原则 [(Ekin, 2026)](https://jet.scholasticahq.com/article/161283.pdf)：

| 原则 | 说明 | 示例 |
|------|------|------|
| 具体优于模糊 | 细节决定输出质量 | 不说"写好一点"，而说"用正式学术语气" |
| 示例优于描述 | 给模型看"怎么做"比说"做什么"更有效 | 提供 1-3 个输入-输出示例 |
| 约束优于放任 | 明确边界比开放提问更可控 | 指定"不超过 200 字""仅回答是或否" |
| 验证优于信任 | 模型输出必须经过核查 | 要求模型标注不确定性或引用来源 |
| 迭代优于一次性 | 提示词工程是持续优化过程 | 测试→调优→再测试的循环 |

### 核心技术

| 技术 | 说明 | 适用场景 |
|------|------|----------|
| 零样本提示（Zero-Shot） | 直接给指令，不提供示例 | 简单、通用的任务 |
| 少样本提示（Few-Shot） | 提供 1-3 个示例引导模型 | 格式严格或复杂任务 |
| 思维链（Chain-of-Thought, CoT） | 要求模型逐步推理 | 数学、逻辑推理等复杂任务 |
| 角色扮演（Role-Playing） | 为模型设定专家身份 | 需要专业领域知识的任务 |
| 任务分解（Task Decomposition） | 将复杂任务拆分为子任务 | 多步骤、流程性任务 |
| 结构化输出（Structured Output） | 指定 JSON/Markdown 等格式 | 需要程序解析输出的场景 |

### Prompt 的四要素框架

一个合格的工程化 Prompt 通常包含四个核心要素 [(51CTO)](https://blog.51cto.com/u_11365839/14597783)：

| 要素 | 位置 | 作用 |
|------|------|------|
| Role（角色） | 开头 | 激活模型相关领域的知识 |
| Task（任务） | 中间 | 明确要完成的具体动作 |
| Context（上下文） | 中间 | 提供任务相关的背景信息 |
| Format（格式） | 结尾 | 约束输出的结构要求 |

将角色放在开头、格式放在结尾，可以有效利用大模型更容易关注上下文开头和结尾信息的特点。

### 提示工程 vs 微调

| 维度 | 提示工程 | 微调（Fine-Tuning） |
|------|----------|---------------------|
| 是否修改模型 | 否，仅优化输入 | 是，更新模型参数 |
| 成本 | 低（只需调整文本） | 高（需要训练数据和算力） |
| 门槛 | 低（任何人可做） | 高（需要机器学习经验） |
| 适用场景 | 快速调整输出质量 | 深度定制模型能力 |
| 效果上限 | 受基础模型能力限制 | 可突破基础模型限制 |

提示工程是所有大模型应用开发的**基础必修课**——一个好的提示词，甚至能让基础模型的输出效果媲美经过简单微调的模型 [(掘金)](https://juejin.cn/post/7626551706944831531)。

### 减少幻觉的策略

LLM 可能生成看似合理但实际错误的内容（幻觉/Hallucination），提示工程提供多种缓解策略：

- **检索增强生成（RAG）**：将外部真实文档注入 Prompt，要求模型"仅基于提供的信息回答"
- **不确定性标注**：要求模型对不确定的信息标注"待验证"
- **多源交叉验证**：让模型从多个角度验证同一信息
- **限制回答范围**：明确告知模型"若不确定，回答'我不知道'"

## 相关概念

- **Prompt（提示词）**：提示工程的操作对象——提示工程是方法论，Prompt 是方法论的输入物。
- **LLM（大语言模型）**：提示工程的目标是最大化 LLM 的输出质量，不修改模型本身。
- **Context Window（上下文窗口）**：提示工程需要在上下文窗口的限制内设计 Prompt，合理分配 Token 预算。
- **RAG（检索增强生成）**：一种结合外部知识检索的提示工程技术，通过在 Prompt 中注入检索结果来减少幻觉。
- **Fine-Tuning（微调）**：与提示工程互补的方法——提示工程不改模型，微调改模型。两者常结合使用。
