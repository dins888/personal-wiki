## 定义

上下文窗口（Context Window）是大语言模型（LLM）在一次推理过程中能够同时"看到"、处理和关联的最大词元（Token）数量。它相当于模型的**短期工作记忆**——就像你一次只能同时记住几页书的内容一样，模型也受限于它能同时关注的文本范围。上下文窗口的大小以 Token 数为单位衡量，它直接决定了模型一次能处理多长的输入、维持多长的对话、分析多大的文档。

### 逐词解释

| 英文 | 词性 | 中文释义 | 在术语中的含义 |
|------|------|----------|----------------|
| Context | 名词 | 上下文、语境、背景 | 指模型推理时可参考的全部前文信息 |
| Window | 名词 | 窗口、视窗 | 指对可见信息范围的限制——只有"窗口内"的内容对模型可见 |

### 组合理解

"Context"与"Window"的组合，形象地表达了一个核心约束：模型虽然拥有庞大的知识储备，但在任何一次推理中，它只能通过一扇"窗口"来看上下文信息。窗口之外的内容，模型根本"看不到"，也就无法利用。这就像通过一扇固定大小的窗户看风景——你能看到窗框内的景色，窗框之外的再美也与你无关 [(IBM)](https://www.ibm.com/think/topics/context-window)。

## 详细说明

### 上下文窗口的内容构成

上下文窗口不仅包含用户看到的对话内容，它实际上承载了四类信息：

| 组成部分 | 说明 | 示例 |
|----------|------|------|
| System Prompt（系统提示词） | 模型的身份设定和行为规则，常驻窗口 | "你是专业客服，仅回答产品问题" |
| 对话历史 | 之前所有轮次的用户输入和模型输出 | 之前的 10 轮问答内容 |
| 当前输入 | 用户本次的提问或上传的文档 | 一份 5 页的 PDF 文档 |
| 模型输出预留 | 模型即将生成的回答所需空间 | 通常预留 10%-20% 窗口空间 |

四项总和不得超过上下文窗口的 Token 上限，超出部分将被截断或遗忘 [(掘金)](https://aicoding.juejin.cn/post/7654022301877157938)。

### 窗口大小与实际容量

| 窗口大小 | 约等于英文词数 | 约等于中文字数 | 典型使用场景 |
|----------|---------------|---------------|-------------|
| 4K Tokens | ~3,000 词 | ~2,000-3,000 字 | 短对话、单轮问答 |
| 8K Tokens | ~6,000 词 | ~4,000-6,000 字 | 短文章处理（GPT-3.5 级别） |
| 32K Tokens | ~24,000 词 | ~15,000-25,000 字 | 单篇论文、中等代码文件 |
| 128K Tokens | ~96,000 词 | ~60,000-100,000 字 | 中篇小说、大型代码库（GPT-4 级别） |
| 200K Tokens | ~150,000 词 | ~100,000 字 | 长篇小说（Claude 级别） |
| 1M+ Tokens | ~750,000 词 | ~500,000+ 字 | 企业知识库、完整项目工程（Gemini 级别） |

> 注意：以上字数为"输入能承载的量"，还需预留 10%-20% 空间给模型输出。

### 为什么窗口不能无限大？

上下文窗口的扩展面临三重约束 [(IBM)](https://www.ibm.com/think/topics/context-window) [(CSDN)](https://blog.csdn.net/m0_63309778/article/details/154742375)：

| 约束 | 原因 | 影响 |
|------|------|------|
| **计算复杂度** | Transformer 自注意力机制的计算量为 O(n²)，Token 数翻倍，计算量翻 4 倍 | 窗口越大，推理越慢 |
| **显存瓶颈** | 每生成一个新 Token 都需参考窗口内所有前文，KV Cache 占用大量显存 | 窗口越大，硬件成本越高 |
| **注意力衰减** | 超长上下文中，模型对中间位置内容的关注度显著下降（"Lost in the Middle"现象） | 窗口大不等于效果好 |

### "Lost in the Middle" 现象

研究发现，LLM 在处理超长上下文时，对**开头和结尾**的信息关注度最高，而**中间位置**的信息容易被"遗忘"或忽略。即使窗口足够大，塞进去 100 万字时，模型对中间内容的提取准确率仍会明显下降。这一现象被称为"Lost in the Middle"（中间迷失效应）[(腾讯云)](https://developer.cloud.tencent.com/article/2666173)。因此，**关键信息应放在窗口的两端**。

### 滑动窗口机制

上下文窗口本质上是**先进先出的滑动队列**——新的对话内容不断进入，最早的内容会被挤出窗口，模型再也"看不到"它们。这意味着：

- 上下文窗口 **≠ 永久记忆**：模型不会记住从第一轮到现在的所有内容
- 长对话中，模型"忘记"之前说的话，不是模型变笨了，而是早期内容已被挤出窗口
- 解决方案包括：定期总结对话历史、使用 RAG 检索外部知识库、管理上下文的优先级

### 扩展上下文的技术

| 技术 | 原理 | 代表模型 |
|------|------|----------|
| RoPE（旋转位置编码） | 通过旋转矩阵编码位置信息，支持长度外推 | LLaMA、Qwen |
| ALiBi（线性偏置注意力） | 用距离惩罚替代位置编码，外推能力强 | BLOOM |
| 滑动窗口注意力 | 只关注附近 Token，复杂度 O(n×w) | Mistral |
| 稀疏注意力 | 选择性关注关键 Token，降低复杂度 | Longformer |

## 相关概念

- **Token（词元）**：上下文窗口以 Token 为单位衡量大小。Token 是窗口内的"计量单位"。
- **LLM（大语言模型）**：上下文窗口是 LLM 的核心参数之一，直接影响模型的能力边界。
- **Prompt（提示词）**：Prompt 占据上下文窗口的空间，过长或过多的 Prompt 会挤压对话历史和模型输出的空间。
- **RAG（检索增强生成）**：当文档超过上下文窗口容量时，RAG 通过检索相关片段再注入窗口，是一种高效的长文本处理策略。
- **KV Cache**：推理过程中缓存注意力计算的键值对，是上下文窗口占用显存的主要来源。窗口越大，KV Cache 越大。
